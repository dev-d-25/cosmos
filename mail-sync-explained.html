How Your Mail App Fetches & Displays Emails

A deep-dive into stubs, enrichment, and why emails disappear or show "unknown sender".

1. Key Concepts

Concept

What is a "Stub"?

A stub is an incomplete record saved to the database. When your app calls Gmail's messages.list API, Gmail only returns { id, threadId } for each email — just identifiers, nothing else. The corsair gmail plugin auto-saves these as stubs:

// gmail/endpoints/messages.ts — messages.list handler
await ctx.db.messages.upsertByEntityId(message.id, {
    ...message,        // only { id: "abc123", threadId: "def456" }
    id: message.id,
    createdAt: new Date(),
});

A stub in the database looks like this:

{
    "id": "abc123",
    "threadId": "def456",
    "createdAt": "2026-06-16T17:00:00Z"
    // NO "from"
    // NO "subject"
    // NO "internalDate"
    // NO "labelIds"
    // NO "payload"
}

When your UI tries to display this stub, it calls toListItem() which looks for d.from → undefined, then looks for d.payload.headers["From"] → no payload → falls back to "" → UI shows "(unknown sender)".

Concept

What is "Enrichment"?

Enrichment is the process of filling in the missing data. After getting IDs from messages.list, your app calls messages.get (with format: "metadata") for each message. This Gmail API call returns the full metadata including headers:

// server/mail/index.ts — syncLabelFromGmail()
for (const id of needsEnrichment) {
    await client.gmail.api.messages.get({
        id,
        format: "metadata",
        metadataHeaders: ["Subject", "From", "To", "Date"],
    });
}

The messages.get endpoint in the gmail plugin then saves a complete record:

// gmail/endpoints/messages.ts — messages.get handler
await ctx.db.messages.upsertByEntityId(result.id, {
    ...result,
    subject: extractSubject(result),
    body: extractBody(result),
    from: extractFrom(result),
    to: extractTo(result),
    id: result.id,
    createdAt: new Date(),
});

After enrichment, the database record looks like this:

{
    "id": "abc123",
    "threadId": "def456",
    "from": "Dev Vaghela <vagheladev25@gmail.com>",
    "subject": "helllo",
    "internalDate": "1750094760000",
    "labelIds": ["INBOX", "UNREAD"],
    "snippet": "helllo",
    "payload": { "headers": [...] },
    "createdAt": "2026-06-16T17:00:00Z"
}

Now toListItem() finds d.from → the email address → UI shows the correct sender name.

3. Bug #1: Emails Disappearing After Refresh

Bug

The Problem: No ORDER BY in the DB query

The db.messages.list() function in kysely/orm.ts has no ORDER BY clause:

// kysely/orm.ts:282-288
list: async (options) => {
    let q = baseQuery(db, accountId, entityTypeName);
    // NO .orderBy() anywhere!
    if (typeof options?.limit === 'number') q = q.limit(options.limit);
    if (typeof options?.offset === 'number') q = q.offset(options.offset);
    const rows = await q.execute();
    return rows.map(parseRow);
},

Without ORDER BY, PostgreSQL returns rows in heap order (roughly insertion order). This means the "first 25" are the 25 oldest-inserted rows, not the 25 most recent emails.

Edge Case

When "(unknown sender)" Still Appears

There are several scenarios where stubs survive without enrichment:

  • Enrichment failure: If messages.get fails for a message (rate limit, network error), Promise.allSettled swallows the error. That message stays as a stub.
  • Partial sync: If Gmail returns 50 messages but only 25 are enriched (the page size), messages 26-50 remain as stubs. If they appear in a later page, they show "(unknown sender)".
  • Stale cache: On refresh, if the cache has enough messages, getMailList serves from cache WITHOUT re-syncing. If some of those cached messages are stubs from a partial previous sync, they show "(unknown sender)".
  • Race condition: The messages.list auto-cache creates stubs, then enrichment updates them. If the DB read happens between these two operations (unlikely but possible with concurrent requests), stubs are returned.

5. The Page Size Mismatch

Additional Issue

SSR fetches 50, Client fetches 25

In page.tsx, the server-side rendering fetches with pageSize: 50:

// page.tsx - getMailPageData()
const [list, profile, labels] = await Promise.all([
    getMailList({ pageIndex: 0, pageSize: 50, ... }),  // 50 items
    ...
]);

But in _client.tsx, the client-side query uses pageSize: 25:

// _client.tsx
const threadsQuery = useMailThreads({
    page: pageIndex,
    pageSize: 25,   // Different from SSR!
    ...
    initialData: initial.list,  // Data from pageSize: 50
});

This means the query keys are different. TanStack Query treats them as separate queries. The SSR data is passed as initialData but the client immediately fetches with pageSize: 25, triggering a new server request.
